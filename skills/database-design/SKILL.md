# Database Design

Schema mistakes compound: code mistakes are refactored in minutes, schema mistakes require migrations of live data. Spend the thinking here.

## Workflow: Model Before You Migrate

For any new app or feature that stores data:

1. List the **entities** (nouns) and **relationships** (1:1, 1:N, M:N) in plain language first. Show the user this list — this is the cheapest moment to catch misunderstanding.
2. For each entity, identify: what uniquely identifies it, what's required vs optional, what can change vs immutable, what gets queried by.
3. Only then write the schema.

## Defaults That Are Almost Always Right

- **Primary keys**: UUID (v4 or v7) or ULID. v7/ULID preferred for index locality. Auto-increment integers are fine for purely internal tables but leak volume and enable enumeration if they ever reach an API.
- **Every table gets** `created_at` (default now) and `updated_at` (trigger or ORM-managed). You will need them; adding later means NULLs forever.
- **Money**: integer minor units (agorot/cents) or `NUMERIC`. `FLOAT` for money is a bug, full stop.
- **Timestamps**: `timestamptz` (Postgres) / UTC always. Timezone conversion is a presentation concern.
- **Text**: `TEXT` over `VARCHAR(n)` in Postgres unless a real constraint exists (n is a business rule, not a guess).
- **Booleans that will grow**: if a boolean might become a third state (`is_published` → draft/scheduled/published), use a `status` text/enum now.
- **NOT NULL by default.** Every nullable column needs a reason: "what does NULL mean here, distinctly from empty/zero?" If no answer, it's NOT NULL with a default.

## Constraints Are Application Logic That Can't Be Bypassed

Put invariants in the database, not only in code — code has bugs, race conditions, and future scripts that skip the service layer:

```sql
email TEXT NOT NULL UNIQUE,
quantity INT NOT NULL CHECK (quantity > 0),
status TEXT NOT NULL CHECK (status IN ('draft','active','archived')),
user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE
```

- Foreign keys always, with a deliberate `ON DELETE` choice: CASCADE for owned children (order → order_items), RESTRICT for referenced masters (don't delete a product that has orders), SET NULL rarely and consciously.
- Uniqueness that's conditional uses partial unique indexes: `CREATE UNIQUE INDEX ON users(email) WHERE deleted_at IS NULL`.
- Check-then-insert in code is a race; UNIQUE + handling the violation error is correct.

## Relationships

- M:N always gets an explicit join table, named after the relationship if it has meaning (`enrollments` not `user_courses`), because it WILL grow attributes (enrolled_at, role, grade).
- Don't store arrays of foreign IDs in a JSON/array column to avoid a join table — you lose FK integrity and query power.
- Polymorphic references (`commentable_type` + `commentable_id`) trade FK integrity for flexibility; prefer separate FKs or separate tables unless variety is truly open-ended.

## JSONB: Where the Line Is

Use JSONB for: genuinely schema-less payloads (webhook dumps, third-party API responses), user-defined custom fields, config blobs read as a whole.
Do NOT use JSONB for: anything you filter/sort/join on regularly, anything with a known stable shape, relationships. "We'll figure out the schema later" JSONB becomes an unqueryable swamp.

## Indexing

- Index every foreign key (Postgres does NOT do this automatically).
- Index columns in frequent WHERE / ORDER BY. Composite indexes: equality columns first, then range/sort column; a composite serves left-prefix queries too.
- Don't pre-index speculatively beyond that — each index taxes every write. Add when a query plan shows a seq scan on a large table.
- Debugging slow queries: `EXPLAIN ANALYZE` first, guess never.

## Migrations Discipline

- Every schema change is a migration file in git, generated by the tool (prisma migrate, drizzle-kit, supabase migration). Direct console edits to a shared/prod DB are forbidden — the schema must be reproducible.
- Migrations on live data are additive-first: add nullable column → backfill → add constraint → (later) remove old. Never rename in one step on a running system; add-copy-swap.
- Destructive migrations (drop column/table) ship at least one release after the code stopped using them.
- Seed data lives in a separate, idempotent script, not in migrations.

## Deletion Strategy

Decide per table, explicitly:
- **Hard delete**: fine for user-owned trivial data; consider what CASCADE takes with it.
- **Soft delete** (`deleted_at timestamptz`): for anything with audit/recovery/legal value. Then EVERY query needs the filter — use ORM global scopes/views so it can't be forgotten — and unique indexes must be partial (see above).
- Personal data + privacy laws: soft delete doesn't satisfy erasure requests; plan a real purge path.

## Multi-Tenancy (SaaS)

Default: shared tables with `tenant_id` (or `org_id`) NOT NULL on every tenant-owned table + composite indexes starting with it. On Supabase/Postgres, enforce with RLS policies on tenant_id so isolation doesn't depend on every developer remembering a WHERE clause. Schema-per-tenant or DB-per-tenant only for hard isolation requirements (compliance, huge tenants) — operational cost is high.

## N+1 and Query Hygiene

- Loading a list then querying per row is the #1 ORM performance bug. Use eager loading (`include`/`with`/joins). If a loop contains `await db.`, stop and restructure.
- Bound every list query (LIMIT) at the data layer even if the API also paginates.
- SELECT the columns you need for hot paths; `SELECT *` on wide tables with JSONB/blob columns wastes real IO.

## Checklist Before Declaring a Schema Done

1. Entities and relationships confirmed with the user in plain language
2. Every table: PK, created_at/updated_at, NOT NULL by default
3. FKs on all references, ON DELETE decided deliberately per relation
4. Invariants as constraints (UNIQUE/CHECK), not only app code
5. FKs and query columns indexed; no speculative index sprawl
6. Migration files generated and committed; reversible or documented why not
7. Deletion strategy decided per table; soft-delete filters enforced structurally
8. Multi-tenant: tenant_id + RLS everywhere applicable
9. Money as integers/NUMERIC; all timestamps UTC
